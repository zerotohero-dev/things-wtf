# 03 · GVK — How Kubernetes Names Things

Every resource in Kubernetes has a three-part identity called a **GVK**
(Group / Version / Kind). Understanding this is essential for reading operator
code, RBAC manifests, and CRD definitions.

---

## The three parts

```yaml
# Group:    spike.io         ← your domain. Core types use "" (empty string)
# Version:  v1alpha1         ← API maturity: alpha → beta → v1
# Kind:     SpikeConfig      ← CamelCase singular name of the type

# The REST path this maps to:
# /apis/{group}/{version}/namespaces/{ns}/{plural-kind}/{name}
# /apis/spike.io/v1alpha1/namespaces/default/spikeconfigs/my-config

# Core types (Pod, Service) are different:
# /api/v1/namespaces/{ns}/pods/{name}
# (no group — they predate the extension API)
```

You'll see GVKs everywhere:

- In CRD manifests (`spec.group`, `spec.versions[].name`, `spec.names.kind`)
- In RBAC rules (`apiGroups`, `resources`)
- In controller-runtime scheme registration
- In `kubectl`: `--api-version=spike.io/v1alpha1 --kind=SpikeConfig`

---

## Resource vs. Kind — a subtle but real distinction

**Kind** is the Go type name: `SpikeConfig`.  
**Resource** is the plural lowercase URL segment: `spikeconfigs`.

They're related but not the same. RBAC rules reference resources; your Go code
references kinds. The CRD maps between them via `spec.names`.

!!! info "Junior context"

    This trips people up when writing RBAC. You write `resources: ["spikeconfigs"]`
    (plural, lowercase), not `resources: ["SpikeConfig"]`. The kubebuilder RBAC
    markers handle this for you automatically when you use `+kubebuilder:rbac:...`
    comments in your controller file.

---

## Versioning is a contract

!!! info "Why versioning matters"

    API versioning isn't just "new features." It's a contract. Once you release
    `v1` of a CRD, you cannot remove fields or change their types without a major
    version bump. The version in the GVK is the signal to clients about what
    guarantees they can rely on:

    - `v1alpha1` — this can change without notice
    - `v1beta1` — breaking changes require a deprecation notice
    - `v1` — stable; no breaking changes, 12-month deprecation minimum

    This matters a lot when other teams write automation against your CRDs.

---

## The Scheme: GVK ↔ Go types

In controller code, a **Scheme** is a registry that maps GVKs to Go struct types
and back. When the client receives a JSON object from the API server, it uses the
Scheme to know which Go struct to deserialize it into.

Every operator registers its types at startup:

```go
// cmd/main.go (generated by kubebuilder)
var scheme = runtime.NewScheme()

func init() {
    // Register standard Kubernetes types (Pod, Service, etc.)
    utilruntime.Must(clientgoscheme.AddToScheme(scheme))

    // Register your custom types.
    // This tells the runtime:
    //   GVK "spike.io/v1alpha1/SpikeConfig" ↔ &spikev1alpha1.SpikeConfig{}
    utilruntime.Must(spikev1alpha1.AddToScheme(scheme))
}

// Generated in api/v1alpha1/groupversion_info.go:
var GroupVersion = schema.GroupVersion{Group: "spike.io", Version: "v1alpha1"}
```

!!! warning "Common mistake"

    Forgetting to register a type in the Scheme causes a panic at startup:
    `no kind is registered for the type X in scheme`, or the controller silently
    never seeing objects of a type you're watching. Every type you `r.Get()` or
    watch must be registered.
